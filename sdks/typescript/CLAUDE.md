# typescript/ -- TypeScript SDK

## Purpose

TypeScript bindings for GoudEngine with two targets: Node.js desktop (via napi-rs)
and web browser (via wasm-bindgen). Published as `@goudengine/sdk` on npm with
conditional exports that select the right backend automatically.

## Structure

```
typescript/
  native/              # napi-rs Rust crate for Node.js
    src/
      lib.rs           # Module root
      game.g.rs        # GoudGame class (codegen)
      entity.g.rs      # Entity class (codegen)
      components.g.rs  # Transform2D, Sprite data structs (codegen)
      types.g.rs       # Vec2, Color, Rect, etc. (codegen)
    Cargo.toml
  src/generated/       # TypeScript source (codegen)
    index.g.ts         # Re-export barrel
    node/index.g.ts    # Node wrapper classes
    web/index.g.ts     # WASM wrapper classes
    web/input.g.ts     # Web input handling
    types/             # Shared TS type definitions
      engine.g.ts
      math.g.ts
      input.g.ts
  wasm/                # wasm-pack output (built from goud_engine crate)
  dist/                # Compiled JS output
  test/smoke.test.mjs  # Smoke tests (node --test)
  index.js             # Root napi binding loader
  index.d.ts           # Root type declarations
```

## Build Commands

```bash
npm run build:native       # Build napi-rs Node addon (.node file)
npm run build:ts           # Compile TS for Node target
npm run build:web          # Build WASM + compile TS for web target
npm run build:all          # Both Node and web
npm run build              # Native + TS (default)
npm run test               # node --test test/*.test.mjs
npm run typecheck          # tsc --noEmit for both targets
```

## f64 vs f32 Design Choice

This is a critical gotcha. napi-rs represents all JavaScript `number` values as
`f64`, but the Rust engine uses `f32` internally. The Node SDK converts between
these at the napi boundary -- `f64` in the JS API, cast to `f32` when calling
engine functions. The WASM target uses `f32` directly since wasm-bindgen supports
it natively. When reading or writing the codegen files, keep this distinction in
mind: Node wrapper code has `f64` casts that the web wrapper does not.

## Codegen

Most files under `native/src/` and `src/generated/` have a `.g.rs` or `.g.ts`
suffix indicating they are generated by the TS SDK codegen tooling (driven by
`codegen/ffi_manifest.json`). Do not hand-edit these files -- modify the codegen
templates or manifest instead.

## Package Exports

The `package.json` uses conditional exports:
- `node` condition loads `dist/generated/index.g.js` (napi-rs addon)
- `browser` condition loads `dist/generated/web/index.g.js` (WASM)
- Explicit `./web` and `./node` sub-paths are also available

## Anti-Patterns

- NEVER hand-edit `.g.rs` or `.g.ts` files -- they will be overwritten by codegen
- NEVER implement game logic in TS -- call through to the Rust engine
- NEVER assume f64 precision matches f32 -- floating point differences can cause
  subtle bugs in collision detection and transform math
