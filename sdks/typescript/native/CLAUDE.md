# native/ -- napi-rs Native Addon

## Purpose

Node.js native addon that bridges the GoudEngine Rust core to the TypeScript SDK.
Built with napi-rs, compiles to a `.node` binary that Node.js loads at runtime.

## Key Files

- `Cargo.toml` -- crate config, depends on `napi`, `napi-derive`, and `goud_engine`
- `build.rs` -- napi-build setup (generates napi registration glue)
- `src/lib.rs` -- module root, wires up the generated modules
- `src/game.g.rs` -- **codegen** GoudGame class with all engine bindings (window, renderer, input, ECS, collision, 3D)
- `src/components.g.rs` -- **codegen** Transform2D, Sprite component data structs
- `src/entity.g.rs` -- **codegen** Entity wrapper
- `src/types.g.rs` -- **codegen** Vec2, Vec3, Color, Rect types and color factory functions

## Codegen

All `.g.rs` files are AUTO-GENERATED by the GoudEngine codegen pipeline. Do not edit them
by hand. Regenerate by running the codegen tool (see `codegen/` in the repo root).

`lib.rs` itself is also generated and just re-exports the generated modules.

## How It Works

1. napi-rs `#[napi]` macros on structs/functions generate N-API bindings at compile time
2. The addon calls `goud_engine::ffi::*` functions directly -- same C-ABI FFI layer used by C#/Python
3. Some ECS operations (add/get/set components) access the context registry directly for efficiency
4. All `unsafe` blocks have `// SAFETY:` comments per project convention

## Building

```bash
cd sdks/typescript
npm run build:native    # Compiles this crate and copies the .node artifact
```

The output is a platform-specific `.node` file loaded by the TypeScript SDK's desktop entry point.

## Gotchas

- `crate-type = ["cdylib"]` -- produces a shared library, not a regular Rust lib
- napi uses `f64` for all numbers (JavaScript has no `f32`), so every FFI call casts `as f32`
- Texture handles are `u64` on the Rust side but passed as `f64` through napi
- LTO is enabled in release profile for smaller binaries
